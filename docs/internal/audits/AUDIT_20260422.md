**Note : 8.6 / 10** (maturité du code) — **Note : 5.2 / 10** (hygiène de livraison).

## 4. Backend Python (Phase 2)

### Findings

#### B1 [P1] `except Exception` : 13 occurrences, toutes justifiées mais fragiles

**Constat** : 13 matches dans 5 fichiers :

- [cinesort_api.py:97](../../cinesort/ui/api/cinesort_api.py#L97) — boundary top-level (log + debug si env var). ✅ justifié
- [cinesort_api.py:802,811](../../cinesort/ui/api/cinesort_api.py#L802) — `# noqa: BLE001`, endpoint maintenance. ✅ justifié mais `noqa` est un *code smell*
- [rest_server.py:436](../../cinesort/infra/rest_server.py#L436) — boundary REST + `M8 : ne pas exposer le message d'exception au client`. ✅ excellent
- [job_runner.py:48,71,254](../../cinesort/app/job_runner.py) — boundary top-level thread. ✅
- [jellyfin_sync.py:96](../../cinesort/app/jellyfin_sync.py#L96) — appel client tiers (`JellyfinError` hérite de `Exception`). ✅
- [apply_support.py:1107,1136,1164,1200,1284](../../cinesort/ui/api/apply_support.py) — 5 catchs post-apply (Jellyfin/Plex refresh, Jellyfin snapshot/restore, boundary apply_changes). ✅

**Impact** : aucun bug latent. Mais la convention AGENTS.md dit « jamais `except Exception` ». L'exception documentée fonctionne, mais un linter plugin pourrait la rendre explicite.

**Recommandation** : définir un `BoundaryError` dans `cinesort/infra/errors.py` et un décorateur `@boundary_handler(log_name)` qui remplace les 13 occurrences. Code plus concis, intention plus claire. **Prioriser P2** — travail de cohérence, pas de bug.

#### B2 [P1] Pas de `requirements-lock.txt` / `requirements.lock`

**Constat** : `requirements.txt` pin en `>=` seulement :
```
pywebview>=5.0
requests>=2.31
segno>=1.6
rapidfuzz>=3.0
```

**Impact** : deux builds sur deux machines différentes peuvent installer des versions différentes. Un `pip install -U` sur le CI peut faire passer un CVE (ex : `requests>=2.32` serait installé même si une vulnérabilité y est découverte).

**Recommandation** : générer `requirements.lock` via `pip freeze > requirements.lock` (ou utiliser `pip-tools`, `uv lock`). Le commit avec pin strict. Le `requirements.txt` reste en `>=` pour expression d'intention.

#### B3 [P2] Ruff lint trop permissif

**Constat** : [pyproject.toml:21](../../pyproject.toml#L21) → `select = ["E9", "F"]`. Ces 2 familles de règles n'attrapent que les erreurs de syntaxe (E9) et les imports / noms inutilisés / undefined (F). **Ignorées** : E501 (line-length 120 mais pas enforced), W (whitespace), B (bugbear : anti-patterns), S (security : eval, assert, hardcoded bind, etc.), N (naming), UP (pyupgrade).

**Impact** : un `except:` nu, un `time.sleep(0)` dans un test, un `hashlib.md5` non documenté → passent.

**Recommandation** : élargir à `select = ["E", "F", "W", "B", "S", "UP"]` et ajouter `ignore = ["E501"]` si line-length 120 non appliqué. Puis `ruff check --fix` pour auto-corriger, mesurer l'ampleur, adresser les manuels.

#### B4 [P2] Pas de type-checker (mypy/pyright)

**Constat** : le code a `from __future__ import annotations` partout et des type hints cohérents. Mais aucun mypy dans `requirements-dev.txt` ni dans `ci.yml`. Les annotations ne sont pas vérifiées.

**Recommandation** : ajouter `mypy --strict` en CI. Commencer par `cinesort/domain/` (cœur pur). Probable : ~50 erreurs à corriger, travail sur 1 journée.

## 5. Frontend web (Phase 3)

### Findings

#### F1 [P1 HAUTE] Skip-link invisible au focus

**Constat** : [web/index.html:14](../../web/index.html#L14) (à vérifier au moment du commit) — `.skip-link { position: absolute; left: -9999px; }` — caché hors écran permanent, pas d'override `:focus`. Le lien `#main-content` (L114) n'est jamais focusable visuellement.

**Impact** : **Viole WCAG 2.4.1 « Bypass Blocks »**. Un utilisateur clavier ne peut pas contourner le sidebar de 220 px.

**Recommandation** : voir diff CSS dans le rapport frontend agent — ajouter `.skip-link:focus { position: fixed; left: var(--sp-4); top: var(--sp-4); background: var(--accent); z-index: 1001; }`.

#### F2 [P1 HAUTE] Absence de Content-Security-Policy

**Constat** : ni `web/index.html` ni `web/dashboard/index.html` ne contiennent de `<meta http-equiv="Content-Security-Policy">`. `rest_server.py` ne renvoie pas non plus de header CSP pour les fichiers statiques.

**Impact** : si un champ dynamique (titre de film depuis TMDb, nom de fichier) arrive dans le DOM via `innerHTML` sans échappement, l'exécution JS est autorisée sans restriction. La présence de 52 `innerHTML` sans escapeHtml visible (sur 206 — rapport frontend) amplifie le risque.

**Recommandation** : ajouter dans `web/dashboard/index.html` (priorité, exposé au LAN) :
```html
<meta http-equiv="Content-Security-Policy"
  content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; font-src 'self'; img-src 'self' data:; connect-src 'self'; frame-ancestors 'none'; base-uri 'self';">
```
Et dans `rest_server.py`, ajouter le header `Content-Security-Policy` sur la réponse `/dashboard/*`.

Dans `web/index.html` (pywebview, moins exposé), idem sauf `connect-src` qui peut inclure `pywebview://`.

#### F3 [P1] 52 instances `innerHTML =` sans `escapeHtml()` immédiatement visible

**Constat** : 206 `innerHTML` totaux dans `web/**/*.js` — 75% utilisent `escapeHtml()`, 25% (52 instances) construisent du HTML sans échappement visible dans le voisinage.

**Impact** : XSS réflexif si la donnée source n'est pas contrôlée. Les titres de films proviennent de TMDb (réputé sûr) ou de l'utilisateur (non-sûr : nom de dossier arbitraire sur le disque).

**Recommandation** : créer un helper `safeBuildHtml(template, vars)` dans `web/core/dom.js` qui échappe tous les `vars` par défaut. Migrer les 52 instances au fur et à mesure. Ajouter un test pour chaque migration.

#### F4 [P2] Chargement de scripts sans `defer` sur desktop

**Constat** : `web/index.html` charge ~43 `<script src="..."></script>` en cascade synchrone. Aucun `defer`, aucun `async`.

**Impact** : Time-to-Interactive ~400 ms vs ~150 ms sur le dashboard (qui utilise ES modules). Acceptable pour pywebview (fenêtre fermée, splash prévu) mais mauvaise pratique.

**Recommandation** : ajouter `defer` sur tous les `<script src>` sauf le bootstrap `app.js`. Tester que l'ordre de dépendances tient (normalement oui car `defer` préserve l'ordre).

#### F5 [P2] Pas de virtualization sur listes > 1000 films

**Constat** : `web/views/validation.js`, `web/views/quality.js` itèrent sur `allRows` et créent 1 `<tr>` par film.

**Impact** : bibliothèque de 2000+ films → 2000 DOM nodes, scroll saccadé, rendu initial ~3 s.

**Recommandation** : si la cible CineSort inclut des bibliothèques > 1000 films, implémenter un windowing simple (200 rows visibles, scroll-linked). Sinon, documenter la limite dans README.

## 7. Sécurité (Phase 5)

### Findings

#### S1 [P0 CRITIQUE] DPAPI fallback silencieux sur OS non-Windows

**Constat** : agent de sécurité rapporte [local_secret_store.py:41-42](../../cinesort/infra/local_secret_store.py) — si DPAPI indisponible (non-Windows ou Windows sans `CryptProtectData`), `_protect_bytes()` retourne `(False, "", message)` **sans exception bloquante**. Les appelants reçoivent potentiellement la chaîne brute dans `settings.json`.

**Impact** : un utilisateur avancé qui clone le projet sous Linux/WSL pour hacker, ou un utilisateur Windows dont DPAPI est corrompu, stocke ses tokens en clair dans `%APPDATA%/CineSort/settings.json`. Le `.gitignore` protège le commit mais pas le disque.

**Recommandation** : (A) refuser de démarrer si DPAPI indisponible avec message bloquant « Ce binaire est Windows uniquement. » ; (B) ou fournir un fallback via `cryptography.fernet` avec clé dérivée de `platform.node() + user SID` (ajoute une dépendance). Option A préférée — CineSort est `onefile` Windows, pas multiplateforme.

#### S2 [P0 HAUTE] REST server bind `0.0.0.0` par défaut

**Constat** : confirmé [rest_server.py:468](../../cinesort/infra/rest_server.py#L468) et `:483` → `host: str = "0.0.0.0"` comme défaut de paramètre. Les commentaires `481-482` documentent le choix : `127.0.0.1` pour pywebview, `0.0.0.0` pour dashboard LAN.

**Impact** : un utilisateur qui active `rest_api_enabled=true` dans les paramètres (pour essayer l'API REST) **expose immédiatement l'API Bearer à tout le LAN**. Si le token par défaut est vide ou faible, tout appareil sur le LAN peut appeler `/api/start_plan`, `/api/apply`, `/api/save_settings`.

**Recommandation** : 
1. Changer le défaut à `host: str = "127.0.0.1"` ligne 468.
2. Ajouter un paramètre `rest_api_public: bool = False` dans les settings. Si true, alors `host = "0.0.0.0"`, sinon `"127.0.0.1"`.
3. Dans l'UI Settings, afficher un avertissement explicite en rouge si `rest_api_enabled=true ET rest_api_public=true` : « L'API est exposée sur le réseau local. Assurez-vous qu'un token fort est configuré. »

#### S3 [P1] CORS `*` permanent

**Constat** : `rest_server.py:212, :477` — `Access-Control-Allow-Origin: *`.

**Impact** : mitigé par l'auth Bearer (l'attaquant doit avoir le token), mais bad practice. Permet à une page malveillante d'envoyer un preflight OPTIONS et de lire les en-têtes.

**Recommandation** : retourner `Access-Control-Allow-Origin: http://{local_ip}:{port}` quand configuré pour LAN, `http://localhost:{port}` pour 127.0.0.1. Ajouter un setting `rest_api_cors_origin` (déjà mentionné dans le code à ligne 477 selon l'agent).

#### S4 [P1] SMTP password stocké en clair (sans DPAPI)

**Constat** : l'agent a identifié que `email_smtp_password` dans `cinesort/ui/api/settings_support.py` est sauvegardé en clair dans `settings.json`, contrairement aux tokens Jellyfin/TMDb/Plex/Radarr qui passent par DPAPI.

**Impact** : réutilisation de mot de passe possible si l'utilisateur met son vrai mot de passe Gmail/Outlook. Fichier `settings.json` peut être synchronisé (OneDrive, backup).

**Recommandation** : ajouter `EMAIL_PASSWORD_FIELD` à la liste DPAPI dans `local_secret_store.py`. Migration douce : au premier démarrage après la fix, détecter un `email_smtp_password` en clair, le chiffrer, écraser.

#### S5 [P1] `innerHTML` sans échappement systématique (52/206)

Voir [§5 F3](#f3-p1-52-instances-innerhtml--sans-escapehtml-immédiatement-visible) et CSP [§5 F2](#f2-p1-haute-absence-de-content-security-policy). Mitigation en double-couche : CSP + `safeBuildHtml()`.

#### S6 [P2] Rate limiter 5/IP/60s contournable par IP-rotation

**Constat** : `rest_server.py:152-192` — rate limit par IP. Sur un LAN, un attaquant avec N appareils peut faire 5N tentatives/60s.

**Recommandation** : ajouter un rate limit **global** en complément : `global_failures`, plafond 20/60s, renvoyer 429 avec message générique. Garder le per-IP en parallèle.

#### S7 [P2] Path traversal message d'erreur révèle le nom de fichier

**Constat** : `rest_server.py:302` (selon agent) — `f"Fichier introuvable: {relative}"` en 404 sur `/dashboard/*` révèle quel fichier a été tenté.

**Recommandation** : réponse générique `{"ok": false, "message": "Fichier introuvable"}`. Logger le chemin complet côté serveur pour audit.

#### S8 [P2] Notifications Win32 : troncature UTF-16 mi-caractère possible

**Constat** : `cinesort/infra/notifications.py:57-61` — `text[:max_chars-1]` avant encode UTF-16LE. Un caractère surrogate pair (emoji, ideogramme > U+FFFF) peut être coupé en deux.

**Recommandation** : après le slice, `text.encode("utf-16-le", errors="ignore").decode("utf-16-le")` pour re-normaliser. Ou utiliser `unicodedata.normalize("NFC", text)` + slice par code points.

#### S9 [P2] Plugin hooks héritent l'environnement parent

**Constat** : `cinesort/app/plugin_hooks.py:135-142` — subprocess sans `env={}`. Le plugin voit `LOCALAPPDATA`, `USERNAME`, potentiellement des secrets si en env var.

**Recommandation** : env minimal explicite : `env={"PATH": os.environ["PATH"], "TEMP": os.environ["TEMP"]}`. Documenter dans `docs/PLUGINS.md` ce qui est disponible.

## 8. Base de données SQLite v10 (Phase 6)

### Findings

#### DB1 [P0 HAUTE] PRAGMA configuration incomplète

**Constat confirmé** : [connection.py:13-15](../../cinesort/infra/db/connection.py#L13) — uniquement `busy_timeout`, `journal_mode=WAL`, `foreign_keys=ON`. Manquent :

- **`PRAGMA synchronous`** : non défini → défaut `FULL`. Sur WAL, `NORMAL` est la valeur canonique (safe + performante). `FULL` annule partiellement le gain WAL.
- **`PRAGMA cache_size`** : non défini → défaut 2000 pages (~8 MB). Une DB avec 10 000 films + perceptual_reports peut dépasser ce cache → thrashing.
- **`PRAGMA temp_store`** : non défini → défaut `DEFAULT` (peut être FILE). `MEMORY` évite les writes temp disk.

**Impact** : les benchmarks (stress tests 1000-5000 dossiers mentionnés dans `tests/stress/`) sous-performent. Les requêtes agrégées (`get_quality_counts_for_runs`, `get_global_stats`) sont plus lentes que nécessaire.

**Recommandation** : patcher `connection.py` :

```python
conn.execute(f"PRAGMA busy_timeout = {busy_timeout_ms}")
conn.execute("PRAGMA journal_mode = WAL")
conn.execute("PRAGMA synchronous = NORMAL")
conn.execute("PRAGMA foreign_keys = ON")
conn.execute("PRAGMA cache_size = -65536")    # 64 MB (signé négatif = KB)
conn.execute("PRAGMA temp_store = MEMORY")
conn.execute("PRAGMA mmap_size = 268435456")  # 256 MB memory-mapped I/O (optionnel)
```

Tester sur `tests/stress/large_volume_flow.py` : attendu 15-30% d'accélération sur les requêtes d'agrégation.

#### DB2 [P1] Cache probe : pas de politique d'invalidation automatique

**Constat** : `probe_cache` (migration 002) clé composite `(path, size, mtime, tool)`. Un fichier rescanné avec des valeurs inchangées → hit. Mais si l'utilisateur supprime un fichier, l'entrée reste en DB indéfiniment.

**Impact** : croissance monotone. Bibliothèque très dynamique → `probe_cache` grossit sans bornes.

**Recommandation** : ajouter une méthode `prune_probe_cache(retention_days=90)` dans `_probe_mixin.py` qui supprime les entrées dont `(now - last_access_ts) > retention`. L'appeler au démarrage ou depuis un bouton « Nettoyer cache » dans les paramètres.

#### DB3 [P2] Pas de table `schema_versions` — `PRAGMA user_version` seul

**Constat** : les migrations 005, 006, 007, 008, 010 incluent `PRAGMA user_version = N`. Version actuelle : 10. Pas de trace de la date ou de l'étape qui a appliqué chaque migration.

**Impact** : debug difficile en prod. `PRAGMA user_version` dit « version 10 » mais pas « migration 007 appliquée le 3 avril à 14h ». En cas de rollback partiel, on ne sait pas où on en est.

**Recommandation** : optionnel. Créer `schema_migrations (version INTEGER PK, applied_at TEXT, sha256 TEXT)` dans une nouvelle migration `011_schema_history.sql`. `migration_manager.py` INSERT une ligne après chaque migration appliquée.

#### DB4 [P1] Tests unitaires absents sur core.py (1319 L)

**Constat** : agent de tests confirme : `cinesort/domain/core.py` (1319 L, cœur métier : Config, Stats, PlanRow, Candidate, NFO, scoring constants, RunStatus, etc.) n'a **pas** de `tests/test_core.py` équivalent.

**Impact** : la logique de Config/PlanRow/sérialisation est uniquement testée indirectement via E2E ou via tests d'autres modules. Coverage 80% probablement atteint via couplage, pas via tests ciblés.

**Recommandation** : créer `tests/test_domain_core.py` avec :
- serialization roundtrip PlanRow → dict → PlanRow
- validation Config (roots dedup, settings clamp)
- calculs de stats agrégées
- edge cases NFO (XML malformé, encodage ASCII/UTF-8/UTF-16)

Estimation : 1 journée pour couverture exhaustive.

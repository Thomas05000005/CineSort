## 1. Contexte et adaptation du scope

L'audit a été demandé avec un template générique orienté SaaS Base44 (rôles, mobile, public links, multi-tenant). CineSort étant une **app desktop Windows mono-utilisateur** distribuée en .EXE PyInstaller, la majorité des dimensions du template ne s'applique pas. Adaptation :

| Phase template | Statut CineSort |
|---|---|
| Roles, multi-tenant, RBAC | N/A (mono-utilisateur Windows) |
| Mobile, viewport 320–768 px | N/A (desktop pywebview, fenêtre redimensionnable) |
| Public links, register flow, password reset | N/A (pas de comptes) |
| Login flow | Limité au **dashboard distant REST** (Bearer token) |
| Permissions | Limitées au **token REST** + ACL fichier OS + DPAPI |
| Notifications email | Présent (SMTP optionnel pour rapports) |
| Intégrations payantes | N/A (TMDb / Jellyfin / Plex / Radarr) |

Tout le reste — CRUD, formulaires, uploads, rapports, perfs, sécurité locale, intégrité données, polish, accessibility — est pleinement applicable.

**Antériorité** : un audit ultra-complet code/architecture existe déjà (`AUDIT_20260422.md`, v7.2.0-dev). Cet audit-ci est une couche **QA produit / launch readiness** complémentaire, qui re-teste les top risques antérieurs sur 7.6.0-dev et couvre les nouveautés v7.5+.

## 2. Résumé exécutif

### Verdict

**Note launch readiness : 78 / 100** — *Almost ready, minor fixes needed*

CineSort est un projet techniquement très mature : architecture 7 couches propres, 1654+ tests unitaires, 121 E2E Playwright, documentation exemplaire. **Deux risques critiques** restent toutefois bloquants pour un usage « vrais clients » :

1. **CR-1 — Atomicité des opérations `shutil.move` non garantie** : un crash ou kill à mi-apply peut laisser la DB et le filesystem dans un état non-réconciliable.
2. **CR-2 — Pas de backup automatique de la base SQLite** : une corruption disque détruit toute la bibliothèque (films matchés, décisions, scores, historique).

Plus 11 findings High dont quatre touchent la sécurité (bind REST, log leakage de clés, DPAPI partiel, idempotence migrations).

### Acteurs

| Acteur | Surface | Authentification |
|---|---|---|

### Workflows critiques (par ordre d'importance pour le launch)

1. **Scan d'un dossier** → `start_plan` → `plan_library()` / `plan_multi_roots()` → écriture `plan.jsonl` + `runs` SQLite.
2. **Validation décisions** → édition manuelle, choix candidats TMDb, save `decisions.json`.
3. **Apply (renommage + déplacement physique)** → `apply()` (dry-run obligatoire d'abord) → `shutil.move()` + journal `apply_operations` SQLite. **Flux le plus dangereux : opérations destructives sur fichiers utilisateur.**
4. **Undo par lot OU par film** → relit `apply_operations`, pre-hash check P1.2, déplacement inverse.
5. **Détection doublons** → `duplicate_support` + `duplicate_compare` (7 critères pondérés).
6. **Quality scoring (CinemaLux)** → ffprobe / mediainfo → tier Platinum / Gold / Silver / Bronze / Reject.
7. **Perceptual analysis (v7.5+)** → ffmpeg + ONNX LPIPS → `perceptual_reports`.
8. **Migration DB au démarrage** → 18 migrations idempotentes avec PRAGMA `user_version`.
9. **Sync intégrations** (Jellyfin / Plex / Radarr) — facultatif mais critique si activé.
10. **Dashboard REST distant** (auth, library, review distant, exports).

## 5. Bugs identifiés — détails

### CRITICAL

#### CR-1 — Atomicité des opérations `shutil.move` non garantie ✅ **RÉSOLU 29 avril 2026** (commit `f34d240`)

- **Composant** : `cinesort/app/apply_core.py:565` — `move_file_with_collision_policy()`, et 6 autres call sites (L409, 703, 1278, 1293, 1333) + `cleanup.py:262`.
- **Reproduction** :
  1. Préparer une biblio de ≥ 100 films à appliquer.
  2. Lancer un apply réel (`dry_run=False`).
  3. Tuer le process pywebview à mi-parcours via Task Manager → End Task, ou simuler BSOD/coupure secteur.
- **Comportement attendu** : soit toutes les opérations passent, soit la DB et le FS restent cohérents (rollback ou marqueur "interrompu" exploitable par undo).
- **Comportement réel** : `shutil.move()` étant non-transactionnel, la moitié des fichiers peut être déplacée sur disque mais `apply_operations` n'a pu écrire que les opérations terminées. Au reboot, l'utilisateur lance un undo → P1.2 hash check sur opérations partiellement écrites peut soit refuser (laisse demi-état), soit restaurer mal (collisions).
- **Cause** : pas de write-ahead log applicatif en amont du move ; pas de marqueur `IN_PROGRESS` côté SQLite avant chaque move.
- **Fix** :
  1. Avant `shutil.move`, écrire ligne `apply_operations` avec `undo_status='PENDING_MOVE'` + checksum source.
  2. Après move, UPDATE `undo_status='SUCCESS'`.
  3. Au démarrage, scanner les `PENDING_MOVE` orphelins et déclencher reconciliation (vérifier si fichier à src, à dst, ou nulle part — agir en conséquence).
- **Impact launch** : risque réel de **perte de cohérence après crash**. Bloquant si l'utilisateur cible peut être interrompu (BSOD Windows fréquents, mises à jour qui rebootent).
- **Implementation livrée** : migration 019 `apply_pending_moves`, helpers `journaled_move()` / `RecordOpWithJournal` / `atomic_move()` dans [cinesort/app/move_journal.py](../../../cinesort/app/move_journal.py), routine [cinesort/app/move_reconciliation.py](../../../cinesort/app/move_reconciliation.py) avec 4 verdicts (completed / rolled_back / duplicated / lost), hook au boot dans [runtime_support.py](../../../cinesort/ui/api/runtime_support.py) + notification UI sur conflits. 9 sites de `shutil.move` wrappés (apply_core + cleanup + 2 undos dans apply_support). 22 tests dédiés + 2 tests existants mis à jour pour la version 19.

#### CR-2 — Pas de backup automatique de la base SQLite ✅ **RÉSOLU 29 avril 2026** (commit `ed1886f`)

- **Composant** : `~/.cinesort/cinesort.db` (chemin standard, à confirmer dans `cinesort/infra/db/connection.py`).
- **Reproduction** :
  1. Scanner 5 000 films, scorer, valider, appliquer.
  2. Corrompre manuellement la DB (truncate fichier, kill pendant WAL checkpoint).
  3. Relancer l'app.
- **Comportement attendu** : restauration depuis backup périodique, ou au minimum message clair « DB corrompue, restaurez depuis X ».
- **Comportement réel** : aucune copie de sécurité automatique. Toute la bibliothèque (films matchés, décisions, historique scores, feedback utilisateur) est perdue. WAL/SHM ne suffisent pas à un crash disque.
- **Fix** : utiliser `sqlite3.Connection.backup()` natif (snapshot online) avant chaque migration et toutes les N applies réussis. Garder les 5 derniers, paramétrable dans Settings.
- **Impact launch** : un seul crash disque = travail de plusieurs heures perdu. Bloquant pour un usage « vrais clients ».
- **Implementation livrée** : nouveau module [cinesort/infra/db/backup.py](../../../cinesort/infra/db/backup.py) avec `backup_db()` natif, `list_backups()`, `rotate_backups()`, `backup_db_with_rotation()`, `restore_backup()` (avec garde-fou). Hook `_backup_before_migrations()` dans `SQLiteStore.initialize()` (skip fresh install). Hook `store.backup_now(trigger="post_apply")` dans `apply_support.apply_changes()` après apply réel. API publique `backup_now()` / `list_db_backups()` exposée pour UI Settings ultérieure. Default rotation : 5 backups dans `<db_dir>/backups/`. 19 tests dédiés.

### HIGH

#### H-1 — Idempotence des migrations SQL ALTER TABLE non garantie

- **Composant** : `cinesort/infra/db/migrations/` — fichiers `013_*.sql` à `018_*.sql` qui font des `ALTER TABLE … ADD COLUMN`.
- **Risque** : SQLite ne supporte pas `ADD COLUMN IF NOT EXISTS` nativement avant 3.35. Si l'app crashe entre `conn.execute(stmt)` et le bump de `PRAGMA user_version`, la re-exécution échoue avec `OperationalError: duplicate column name`.
- **Fix** : encapsuler chaque ALTER dans une garde `PRAGMA table_info(t)` Python-side, ou try/except `OperationalError` sur la chaîne « duplicate column ».
- **Impact** : un user qui crashe pile pendant la migration au 1er upgrade perd l'app jusqu'à manipulation manuelle.

#### H-2 — Disque plein non géré explicitement pendant apply

- **Composant** : `cinesort/app/apply_core.py:405` — `move_to_review_bucket()` et tout site d'appel à `shutil.move`.
- **Reproduction** : remplir le disque cible à > 99 %, lancer apply.
- **Fix** : avant chaque batch, calculer somme `os.path.getsize(src)` et comparer à `shutil.disk_usage(dst).free` avec marge 10 %. Refuser l'apply avec message clair sinon.
- **Impact** : apply interrompu en plein milieu = retombe dans CR-1.

#### H-3 — Clés API potentiellement loggées en clair via `exc_info=True`

- **Composant** : tout module qui fait `logger.error("…", exc_info=True)` autour d'un appel TMDb / Jellyfin / Plex / Radarr.
- **Risque** : la traceback inclut souvent l'URL complète, qui peut contenir la clé en query string (TMDb v3 utilise `?api_key=…`). Idem pour les headers Jellyfin (`Authorization: MediaBrowser Token="…"`).
- **Fix** :
  1. Vérifier que le client TMDb passe la clé en header (mode v4) plutôt qu'en query (v3).
  2. Ajouter un `logging.Filter` qui scrub `api_key=`, `Authorization:`, `X-Plex-Token:`, `X-Api-Key:` dans tous les messages.
- **Impact** : si l'utilisateur partage un log (support, debug, GitHub issue), sa clé fuit.

#### H-4 — REST server bind `0.0.0.0` par défaut (à reconfirmer sur 7.6.0-dev)

- **Composant** : `cinesort/infra/rest_server.py` — l'audit du 22 avril mentionne L468 et L483.
- **Risque** : si l'utilisateur active le dashboard distant sans comprendre, l'API est accessible sur tout le LAN (et potentiellement Internet si NAT mal configuré). Le Bearer token protège mais bruteforceable si court.
- **Fix** :
  1. Bind `127.0.0.1` par défaut, exiger un toggle explicite « Exposer sur le réseau local » qui bind `0.0.0.0` avec warning rouge dans l'UI.
  2. Forcer un token de ≥ 32 caractères aléatoires.
  3. Logger chaque tentative d'auth échouée + rate limit agressif (déjà 5/IP/60s — garder).
- **Impact** : variable selon environnement. Pour un cinéphile chez lui, faible. Pour un colocataire / bureau partagé, élevé.

#### H-5 — Pas de virtualisation des tables UI sur grandes biblios

- **Composant** : `web/views/validation.js`, `web/views/library.js`, `library-v5.js`.
- **Reproduction** : importer une biblio de 5 000+ films, ouvrir la table validation.
- **Comportement** : rendu DOM complet de toutes les rows → freeze UI 5–30 s, scroll saccadé.
- **Fix** : virtualisation (clusterize.js ou implémentation maison via `IntersectionObserver`).

#### H-6 — Logs sans rotation ✅ **RÉSOLU 29 avril 2026** (commit `80ca098`)

- **Composant** : `ui_log.txt` dans chaque `run_dir`, plus loggers Python par défaut sans `RotatingFileHandler`.
- **Risque** : sur un usage long (mois), occupation disque indéfinie.
- **Fix** : `RotatingFileHandler(maxBytes=50MB, backupCount=5)`.

#### H-7 — `ffmpeg` absent provoque exception non-gracieuse côté perceptuel ✅ **RÉSOLU 29 avril 2026** (commit `ca53cda`)

- **Composant** : `cinesort/domain/perceptual/ffmpeg_runner.py:39` — `resolve_ffmpeg_path()`.
- **Reproduction** : désinstaller ffmpeg, lancer une analyse perceptuelle.
- **Fix** : try/except ciblé qui retourne un `PerceptualResult` partiel avec verdict `unavailable: ffmpeg manquant`, et propose le bouton « Installer ffmpeg » déjà présent dans Settings.

#### H-8 — Texte anglais dans une UI 100 % FR ✅ **RÉSOLU 29 avril 2026** (commit `236d643`)

- **Composant** : `web/index.html` ligne 449 — `<button id="btnUndoV5Preview">Preview selection</button>`.
- **Fix** : « Aperçu de la sélection ».

#### H-9 — `onclick` inline sur `#btnQualitySimulate` assume une fonction globale ✅ **RÉSOLU 29 avril 2026** (commit `236d643`)

- **Composant** : `web/index.html` ligne ~470 — `onclick="if (typeof openQualitySimulator === 'function') openQualitySimulator();"`.
- **Risque** : silent fail si la fonction n'est pas chargée (race condition au boot).
- **Fix** : event listener côté JS dans `web/views/quality-simulator.js`.

#### H-10 — Email report SMTP : mot de passe potentiellement non protégé par DPAPI ✅ **DÉJÀ RÉSOLU sur 7.6.0-dev** (vérifié 29 avril)

> Le finding antérieur (22 avril, v7.2.0-dev) était obsolète : `_persist_protected_secret(legacy_field="email_smtp_password", secret_field=SMTP_PASSWORD_SECRET_FIELD, ...)` dans `settings_support.py:409` route déjà le mot de passe SMTP via DPAPI comme les autres clés.

- **Référence** : audit antérieur §7 mentionne « DPAPI appliqué aux clés TMDb/Jellyfin/Plex/Radarr avec un trou sur le mot de passe SMTP ».
- **Composant** : `cinesort/app/email_report.py` (à vérifier).
- **Fix** : router le SMTP password à travers `local_secret_store.protect_bytes()` comme les autres.

#### H-11 — Atomicité Jellyfin watched-state restore après refresh ✅ **RÉSOLU 29 avril 2026** (commit `9a2d125`)

- **Composant** : `cinesort/app/jellyfin_sync.py` — snapshot AVANT apply, restore APRÈS refresh + polling.
- **Risque** : si le polling timeout avant que Jellyfin ait fini de re-indexer, l'état « vu » n'est pas restauré → impression de perte d'historique côté Jellyfin (cosmétique mais visible).
- **Fix** : timeout configurable, retry exponentiel, état persistant en DB pour reprendre après crash.

### Vague 1 — AVANT exposition publique (~1–2 jours)

# Internationalisation (i18n) â€” guide developpeur

> Infrastructure i18n introduite en Vague 6 du plan **Polish Total v7.7.0**
> (R4-I18N-1 a R4-I18N-6). Locales supportees : **fr** (defaut) et **en**.

## Architecture

```
locales/
  fr.json                # Source de verite (FR = canon)
  en.json                # Traductions (mirror strict des cles top-level)

cinesort/domain/
  i18n_messages.py       # Backend Python : t() / set_locale() / get_locale()

web/dashboard/core/
  i18n.js                # Frontend JS : t() / setLocale() / onLocaleChange()
  format.js              # Formatters Intl.* (dates, nombres, tailles)

cinesort/ui/api/
  cinesort_api.py        # Endpoint REST POST /api/set_locale
  settings_support.py    # Setting `locale` persiste, validation _normalize_locale
```

## Convention des cles

Format hierarchique : `category.subcategory.detail`.

```jsonc
{
  "common": {
    "cancel": "Annuler"
  },
  "settings": {
    "saved_at": "Sauvegarde a {{time}}"   // interpolation : {{var}}
  },
  "qij": {
    "quality": {
      "tab_title": "Qualite"               // 3 niveaux ok
    }
  }
}
```

Categories top-level officielles (V6-01 a V6-05) :
`common`, `errors`, `notifications`, `settings`, `qij`, `library`,
`processing`, `glossary`, `help`, `sidebar`, `topbar`, `danger_zone`.

## Ajouter une nouvelle string

1. **Choisir la cle** sous une categorie existante (ex.
   `settings.naming.preset_label`). Si nouvelle categorie justifiee : creer une
   entree top-level dans **fr.json ET en.json** (sinon
   `test_top_level_categories_strict_mirror` echoue).

2. **Ajouter la valeur FR** dans `locales/fr.json`.
3. **Ajouter la valeur EN** dans `locales/en.json` (meme cle, traduction).
4. **Utiliser dans le code** :
   - Python : `from cinesort.domain.i18n_messages import t` puis
     `t("settings.naming.preset_label")`.
   - JS : `import { t } from "../core/i18n.js";` puis
     `t("settings.naming.preset_label", { count: 5 })`.

5. **Lancer les tests** :
   ```
   python -m unittest tests.test_i18n_round_trip tests.test_i18n_translations
   ```

## Ajouter une nouvelle locale (ex. "es")

1. Etendre `SUPPORTED_LOCALES` dans :
   - `cinesort/domain/i18n_messages.py` (constante module)
   - `cinesort/ui/api/settings_support.py` (constante `SUPPORTED_LOCALES`)
   - `web/dashboard/core/i18n.js` (constante `SUPPORTED_LOCALES`)
2. Creer `locales/es.json` en copiant `fr.json` puis traduire.
3. Mettre a jour les tests :
   - `test_supported_locales_unchanged` -> ajuster le set attendu
   - `test_top_level_categories_strict_mirror` -> ajouter `es` au scan
4. Verifier que `i18n_messages._resolve_locales_dir()` resout bien le fichier
   en bundle PyInstaller (deja generique).
5. Tester : `set_locale("es")` puis `t("common.cancel")` retourne la valeur ES.

## Limitations connues (post Vague 6)

- **Vues legacy non i18n** : `runs.js`, `quality-simulator.js`, parties de
  `library/*` contiennent encore des chaines FR hardcodees. Le test
  `test_report_french_candidates_in_dashboard_views` les liste (informatif,
  ne fail pas tant que le total reste sous 3000).
- **Backend partiellement i18n** : V6-03 a porte les ~70 sites les plus
  visibles (errors + notifications). Reste ~60 chaines suspectes dans
  `cinesort/ui/api/*.py` (cf. `test_report_french_candidates_in_api_layer`).
- **Formatters Intl.*** : `format.js` (V6-04) couvre dates, nombres, tailles
  octets, pourcentages, durees. Les autres helpers de formatage ad-hoc
  restent en `fr-FR` hardcode.
- **EN plus riche que FR** : V6-05 a ajoute glossary (38 termes) + FAQ (15
  entrees) + commons additionnels en EN. ~113 cles sont presentes en EN sans
  equivalent FR. Sera resorbe par retro-portage Vague 7.
- **Fallback silencieux** : si une cle existe en FR mais pas en EN,
  `t(key)` retombe sur la valeur FR (volontaire pour livrer EN incomplet sans
  casser l'UI). Pas de warning runtime â€” surveille via les tests de parite.
- **Interpolation strictement `{{var}}`** : pas de pluriels (ICU MessageFormat
  non supporte), pas de selecteurs genre/contexte. Pour pluriels :
  utiliser deux cles distinctes `errors.one_file` et `errors.many_files`.
